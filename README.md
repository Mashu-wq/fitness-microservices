# Fitness AI Microservices

A cloud-native fitness tracking platform built with Spring Boot and Spring Cloud. Users log activities and automatically receive AI-powered recommendations generated by Google Gemini — analyzing performance, suggesting improvements, and highlighting safety precautions.

---

## Architecture

```
Client
  │
  ▼
API Gateway (8080)  ◄── Keycloak OAuth2/JWT validation
  │                      └── Auto-registers new users (X-User-ID header injection)
  ├──► User Service (8081)        MongoDB: fitnessuser
  ├──► Activity Service (8082)    MongoDB: fitnessactivity
  │         │
  │         │ RabbitMQ (fitness.exchange → activity.queue)
  │         ▼
  └──► AI Service (8083/8084)     MongoDB: fitnessrecommendation
              └── Google Gemini API
```

### Microservices

| Service | Port | Responsibility |
|---|---|---|
| **Eureka** | 8761 | Service discovery & registration |
| **Config Server** | 8888 | Centralized configuration |
| **API Gateway** | 8080 | Auth, routing, user sync |
| **User Service** | 8081 | User profiles & validation |
| **Activity Service** | 8082 | Activity tracking & publishing |
| **AI Service** | 8083/8084 | AI recommendations via Gemini |
| **RabbitMQ** | 5672 / 15672 | Async messaging |

### How It Works

1. **Authentication** — All requests go through the API Gateway, which validates JWTs issued by Keycloak. On first request, the `KeycloakUserSyncFilter` extracts the user's Keycloak ID and auto-creates a local user record in MongoDB, then injects an `X-User-ID` header for downstream services.

2. **Activity Tracking** — A user posts a workout to `POST /api/activities`. The Activity Service saves it to MongoDB and publishes an event to RabbitMQ (`fitness.exchange`, routing key `activity.tracking`).

3. **AI Recommendation** — The AI Service listens on `activity.queue`, picks up the event, builds a structured prompt, and calls the Google Gemini API. The response is parsed for analysis, improvements, workout suggestions, and safety notes — then persisted to MongoDB.

## Tech Stack

- **Java 21** · Spring Boot 3.4.5 · Spring Cloud 2024.0.1
- **Spring Cloud Gateway** (reactive/WebFlux) — API gateway
- **Spring Cloud Netflix Eureka** — service discovery
- **Spring Cloud Config** — externalized configuration
- **Spring Security OAuth2** + **Keycloak** — authentication & authorization
- **Spring Data MongoDB** — persistence (one database per service)
- **Spring AMQP** + **RabbitMQ** — async event messaging
- **Google Gemini API** — AI-generated fitness recommendations
- **Docker / Docker Compose** — containerized deployment
- **Lombok** · **SpringDoc OpenAPI** (Swagger on AI Service)

## Keycloak Setup

1. Start Keycloak at `http://localhost:7080`
2. Create (or use the default) realm: `master`
3. Create an OAuth2 client named `fitness-app-gateway` with:
   - Grant type: `authorization_code`
   - Redirect URI: `http://localhost:8080/login/oauth2/code/keycloak`
   - Scopes: `openid`, `profile`, `email`

## Running the Application

### With Docker Compose (recommended)

```bash
cd fitness-microservice

# Set required environment variables for AI Service
export GEMINI_API_URL=https://generativelanguage.googleapis.com/v1beta/models/gemini-pro:generateContent
export GEMINI_API_KEY=your_gemini_api_key

docker-compose up --build
```

Startup order is enforced: Eureka → Config Server → Gateway + UserService + ActivityService + AIService.

### Running a Service Locally

```bash
cd fitness-microservice/<service-name>
mvn spring-boot:run
```

Ensure Eureka and Config Server are already running before starting dependent services.

## API Endpoints

All routes are accessed through the gateway at `http://localhost:8080`. A valid Keycloak JWT must be included in the `Authorization: Bearer <token>` header.

### User Service — `/api/users`

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/users/register` | Register a new user |
| `GET` | `/api/users/{userId}` | Get user profile |
| `GET` | `/api/users/{userId}/validate` | Validate user existence |

### Activity Service — `/api/activities`

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/activities` | Log a new activity (triggers AI pipeline) |
| `GET` | `/api/activities` | Get all activities for current user |
| `GET` | `/api/activities/{activityId}` | Get a specific activity |

**Example request body for logging an activity:**
```json
{
  "type": "RUNNING",
  "duration": 45,
  "caloriesBurned": 400,
  "startTime": "2025-07-11T07:00:00",
  "additionalMetrices": {
    "pace": "5:30/km",
    "heartRate": "145 bpm",
    "distance": "8.1 km"
  }
}
```

### AI Service — `/api/recommendations`

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/recommendations/user/{userId}` | Get all recommendations for a user |
| `GET` | `/api/recommendations/activity/{activityId}` | Get recommendation for an activity |

Swagger UI is available at: `http://localhost:8083/swagger-ui.html`

## AI Recommendation Structure

When an activity is logged, Gemini returns a structured analysis saved as:

```json
{
  "recommendation": "Overall: ... \nPace: ... \nHeart Rate: ... \nCalories Burned: ...",
  "improvements": [
    "Pace: Increase cadence to reduce injury risk"
  ],
  "suggestions": [
    "Interval Training: Alternate 1 min fast / 2 min easy for 20 minutes"
  ],
  "safety": [
    "Warm up for at least 5 minutes before intense effort",
    "Stay hydrated throughout your run"
  ]
}
```

## Configuration

All service configurations are centrally managed by the Config Server at `fitness-microservice/configserver/src/main/resources/config/`:

| File | Service |
|---|---|
| `api-gateway.yml` | Gateway routes, Keycloak OIDC config |
| `user-service.yml` | MongoDB URI, port |
| `activity-service.yml` | MongoDB URI, RabbitMQ config |
| `ai-service.yml` | MongoDB URI, RabbitMQ, Gemini API |

RabbitMQ exchange/queue settings (shared between Activity and AI services):
- Exchange: `fitness.exchange`
- Queue: `activity.queue`
- Routing Key: `activity.tracking`
